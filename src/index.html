Architecture

⚡

gRPC Transport Layer

Binary Protocol Buffers over HTTP/2 for minimal overhead. The RateLimitService.Check RPC handles every inbound rate-check request with typed request/response contracts.

grpcprotobuf
🔴

Redis Atomic Backend

All counter increments and token bucket updates run as Lua scripts inside Redis — making every operation truly atomic, eliminating race conditions in concurrent multi-instance deployments.

Lua scriptsthread-safe
🏭

Strategy + Factory Patterns

The IRateLimitStrategy interface decouples algorithm logic from the server layer. RateLimitFactory constructs the right implementation at startup based on TOML config — new algorithms require zero server changes.

StrategyFactory
⚙️

Dynamic TOML Configuration

Every endpoint maps to an independent rule — strategy type, limit, and interval — all defined in config.toml. No recompilation needed to reconfigure rules.

tomlplusplus
🐳

Containerised Production Stack

Multi-stage Dockerfile targeting Debian Slim for a minimal footprint. Docker Compose orchestrates the app container and Redis together with a single command. The gRPC server binds to 0.0.0.0:50051 and reads config from the mounted volume.

Rate Limiting Algorithms

01

Fixed Window

Divides time into fixed intervals and counts requests per window. Simple and memory-efficient — a counter key is set with a TTL equal to the window size. At window boundary, the key expires and resets automatically.

Best for: API rate limiting with clear time boundaries
02

Token Bucket

Tokens refill at a steady rate up to a maximum capacity. Bursts are allowed up to the bucket size, smoothing out traffic spikes while maintaining a long-term average rate. State persisted in Redis.

Best for: Endpoints needing burst tolerance
